# 量化交易系统 (QuantSystem)

## 项目概述
Python 量化交易策略回测与实盘系统

## 技术栈
- Python 3.13
- pandas, numpy, pytest
- 数据源: eastmoney, sina, akshare, tushare, baostock

## 目录结构
- `data/` — 数据获取与清洗模块
- `backtest/` — 回测引擎
- `factors/` — 因子计算
- `sniper/` — 狙击手架构 V3（L0-L4 四层选股 + 回测引擎）
- `scripts/` — 执行脚本
- `tests/` — 测试用例
- `docs/` — 设计文档（见下方文档版本规则）

## 开发规范
- 使用 `black` 格式化代码
- 测试用 `pytest`，要求覆盖率 > 80%
- 所有数据源调用必须走 `DATA_SOURCE_PREFERENCE` 自动降级
- 禁止硬编码单数据源（特别是指数数据）
- 禁止提交 secrets 或 .env 文件

## 独立评审闭环法则

> 同步自 `CLAUDE.md §6`，Codex 项目同样适用。

1. **发现-修复-再验证三阶段分离**：评审发现问题后，修复行为与修复验证必须由不同的独立主体执行。自审自改不视为有效验证。
2. **虚假注释禁令**：严禁在代码中添加声称修复了某问题但未实现实质逻辑的注释。此类"注释式修复"比不修更差。
3. **多人多角度评审**：评审应至少从 3 个独立角度（正确性、架构、安全/陷阱）展开。
4. **循环终止条件**：修复-再验证循环持续直到独立验证给出"全部 TRULY FIXED"的结论。
5. **全链路数据测试**：代码通过评审后，必须执行一次完整的端到端数据测试，确认零报错后方可合并。

### 2. 代码变更必须同步文档规则

> **每次代码变更后，必须在 24 小时内更新对应的设计文档。**

违反此规则的后果：

```
代码改了但文档没更新 → 后续开发按旧文档实现 → 代码与设计进一步偏离 → 更多 bug → 更多时间修复
```

具体执行：

| 代码变更类型 | 需要更新文档 |
|-------------|-------------|
| 新增/修改模块 | 更新设计文档的模块描述（函数签名、参数、返回值、设计逻辑） |
| 修改核心逻辑 | 更新设计文档的设计描述（公式、阈值、条件） |
| 新增配置参数 | 更新设计文档的配置表 |
| 废弃/删除模块 | 文档中标记为 DEPRECATED，而非直接删除 |
| 参数调优 | 更新文档中的参数表，标注修改日期 |

### 3. 任务开始前的文档核对步骤

> **每个新任务开始前，先确认设计文档是否为最新。**

步骤：
1. 读取对应模块的设计文档
2. 读取对应模块的代码文件
3. 逐条对比文档描述与实际代码
4. 如有偏差：
   - 偏差 < 20%：立即更新文档再继续
   - 偏差 >= 20%：**先更新整个文档再开始**（不可只修一处）

### 4. 文档更新优先级

| 优先级 | 场景 | 更新范围 |
|--------|------|---------|
| P0 | 核心逻辑变更 | 函数签名 + 算法描述 + 配置表 |

### 5. 旧文档处理规则

## 数据安全规则（CRITICAL — 2026-06-06 新增）

### 1. 数据删除/修改前必须确认

**任何涉及数据删除、批量修改、表结构变更的操作，必须经用户确认后才能执行。**

包括但不限于：
- `DELETE` / `DROP` / `ALTER` SQL 语句
- 删除或覆盖 parquet / CSV / 缓存文件
- 批量 `UPDATE` 数据库行
- 清空表或数据库

### 2. 代码中删除操作必须加注释标记

所有含数据删除/修改逻辑的代码行，上方必须加明确注释：

```python
# ⚠️ DATA_MODIFICATION — 删除重复行（确认：可删除，不影响有效数据）
cursor.execute("DELETE FROM daily_bars WHERE date LIKE '% %'")
```

注释格式：`# ⚠️ DATA_MODIFICATION — <操作说明>（确认：<确认理由>）`

### 3. 操作前检查清单

在执行任何数据修改前，依次确认：
1. ❓ 是否已获用户明确授权？
2. ❓ 修改范围是否可回滚？（建议先备份或加 `WHERE` 限制）
3. ❓ 影响行数/数据量是否已预估？
4. ❓ 是否有替代方案（如软删除/标记禁用而非直接删除）？

未通过上述检查 → 先问用户。

## 效率准则（血泪教训总结）

### 1. 测试用 mock 数据，不要用全量回测

修数据提取逻辑时：

```
错误做法: 修逻辑 → 跑 30 组回测(35min) → 看结果 → 再修
正确做法: 修逻辑 → 造 3 条 mock 交易数据测 extract_trades(0.1s) → 确认逻辑正确 → 最后跑一次全量
```

mock 数据直接构造模拟的 `BacktestEngine` 返回值，覆盖：正常 SELL、BUY（应过滤）、pnl=None（应过滤）、日期格式异常、空字段等边界。

**99% 的 bug 在 mock 阶段就能发现，不需要跑回测。**

### 2. 一次全链路溯源，不要逐层碰运气

只分析当前层会反复漏问题：

```
发现问题: pnl_pct=0 → 修 extract_trades 加 fallback
又发现: hold_days=0 → 再修
又发现: BUY 噪声 → 再过滤
```

正确做法：**先溯源到数据源头（查引擎/API/数据库的输出 schema），一次发现所有缺失字段，再在下游做统一防御。不再从中间层开始猜。**

### 3. 数据层代码质量纪律

**规则 1：写完一个数据通路，5 分钟内验证它**

每个数据通路（fetch → normalize → store → read）完成后，立即用 mock 数据验证写入和读回，不等到"全部完成"再测。

本轮最大教训：写了 Fetcher + Normalizer + warehouse + scripts 共 13 个文件后才跑测试，导致 store 的列不匹配 bug 隔了 30 分钟才发现。

**规则 2：改 schema 必须同时验证写入代码**

改 DDL（加列/改类型）后，立即测试 store 函数能否正确处理新列。`ensure_schema()` 加了列但 `store_daily_bars()` 没跟着对齐 → 数据写入静默丢失 18420 行。

**规则 3：跨平台功能必须在目标平台验证**

Windows spawn + multiprocessing 必须在 Windows 上实际跑一次。准用但不准假设，worker 函数不准在 `__main__` 中定义，`__file__` 不准在子进程中依赖。

### 4. 部署操作纪律

> 上次事故：前端首页无法显示。4 轮修复尝试全部失败，反而导致本地源码损坏、服务器文件缺失。
> 根因：PowerShell 多行命令截断 + 没有版本控制 + 直接在服务器操作。

**规则 1：多文件传输永远用 tar 包，不要用空格分隔的文件列表**

```powershell
# ❌ 错误 — 空格分隔多文件，PowerShell 换行导致参数截断，覆盖本地文件
scp -i key src/ package.json vite.config.js index.html root@ip:/path/

# ✅ 正确 — tar 打包成一个文件再 scp
tar czf $env:TEMP\deploy.tar.gz --exclude=node_modules src/ package.json vite.config.js
scp -i key $env:TEMP\deploy.tar.gz root@ip:/path/deploy.tar.gz
```

**规则 2：不要在 PowerShell 中使用 `&&` 连接多步 SSH 命令**

```powershell
# ❌ 错误 — && 在 PowerShell 中粘贴多行时会截断为独立命令
ssh key root@ip "cd /path && rm -rf src && tar xzf file.tar && npm run build"

# ✅ 正确 — 用 ; 替代 &&，或每步独立执行
ssh key root@ip "cd /path; rm -rf src; tar xzf file.tar; npm run build"

# ✅ 更安全 — 每步独立确认
ssh key root@ip "echo 1. 连接成功"
# 确认上一步成功后再执行下一步
```

**规则 3：永远不要在服务器上直接 sed 修改构建产物**

```bash
# ❌ 错误 — 修改 dist/ 中的 JS 文件（sed 修改后 dist 与源码不一致）
sed -i 's/old/new/g' /path/dist/assets/index-*.js

# ✅ 正确 — 修改源码后重新 npm run build
```

**规则 4：修改后先在本地构建验证，再部署到服务器**

无论何时修改前端代码：
```
cd frontend && npm run build    # 确认本地构建成功
```
只有构建成功，才上传到服务器。如果文件损坏，本地 `npm run build` 会先报错。

**规则 5：项目必须初始化 Git 版本控制**

没有 git 就无法回滚。一次错误的 scp 命令就导致文件永久损坏。恢复只需要一行：
```bash
git checkout package.json
```

**规则 6：复杂操作分段确认，不要一步到位**

```powershell
# ❌ 错误：一步到位，失败后无法定位
步骤A && 步骤B && 步骤C

# ✅ 正确：每步独立执行、独立确认
Write-Host "步骤 1..."
# ... 执行并检查结果
Write-Host "步骤 2..."
# ... 确认无误后继续
```
