# 项目规则：设计文档与代码同步

> 这是从 QuantSystem 项目教训中提炼的规则，适用于所有后续项目。

---

## 问题背景

QuantSystem 项目反复出现：

```
设计文档说 A → 代码实现 B → 后续开发按 A 做 → 又发现需要按 B 修
           ↓
    循环往复，永远在修 bug，永远在补功能
```

根因：**设计文档与代码不同步，且没有机制约束同步。**

---

## 规则

### R1: 唯一真相源（Single Source of Truth）

每个系统模块必须有且仅有 **一个** 设计文档作为当前真相源。

```
❌ BAD:
  设计文档: factor-design-v2.md (V2.12)
  对照文档: 代码实际需求-vs-设计V2对照.md
  架构文档: 三层统一评分架构设计.md
  → 三份文档互相矛盾，谁都不知道哪个是最新的

✅ GOOD:
  唯一真相源: 狙击手架构V3-设计文档.md (标记 ACTIVE)
  旧文档: factor-design-v2.md (标记 DEPRECATED)
  旧文档: 代码实际需求-vs-设计V2对照.md (标记 DEPRECATED)
```

### R2: 代码变更 = 文档变更

> **每次代码变更后 24 小时内必须更新对应文档。**

| 代码变更类型 | 文档更新范围 | 耗时 |
|-------------|-------------|------|
| 修改核心逻辑（算法/公式/条件） | 更新设计描述 | ~5 min |
| 修改函数签名/接口 | 更新函数签名表 | ~2 min |
| 新增功能 | 新增模块描述 | ~10 min |
| 参数调优 | 更新参数表，标注日期 | ~1 min |
| 新增/修改配置 | 更新配置表 | ~1 min |
| 代码重构（逻辑不变） | 不更新 | 0 min |

### R3: 任务开始前必须核对文档

```
开始任何开发任务前:
  1. 读取设计文档
  2. 读取对应代码
  3. 对比
  4. 有偏差 → 先修文档再继续
```

这不是"可选步骤"，是 **硬性前置条件**。

### R4: 文档标记规范

| 标记 | 含义 | 用法 |
|------|------|------|
| `**ACTIVE**` | 当前最新，唯一真相源 | 每个模块一个 |
| `**DEPRECATED**` | 已废弃，禁止参考 | 旧版本文档 |
| `**IN_PROGRESS**` | 开发中，可能不完整 | 新模块设计 |
| `✅ 已实现` | 设计中有，代码已实现 | 标注在差距表中 |
| `❌ 未实现` | 设计中有，代码未实现 | 标注在差距表中 |
| `⚠️ 偏差` | 设计说的和代码做的不一样 | 标注在差距表中 |

### R5: 偏差处理流程

当发现设计与代码存在偏差时：

```
发现偏差
  ↓
判断偏差类型:
  ├─ 代码是对的，文档过时 → 更新文档 (最快路径)
  ├─ 文档是对的，代码有 bug → 修改代码
  └─ 设计本身有问题 → 修改设计 + 修改代码
  ↓
更新文档后，标注修改日期和修改原因
  ↓
在文档底部添加 "变更日志" 条目
```

### R6: 变更日志（Changelog）

每个设计文档底部必须有变更日志：

```markdown
## 变更日志

| 日期 | 版本 | 变更内容 | 原因 |
|------|------|---------|------|
| 2026-06-03 | V3.0 | 完全重写，从代码反推 | 旧文档已废弃 |
| 2026-06-03 | V3.1 | 新增 L2 向量化设计 | 性能优化 Phase 1 |
```

### R7: 定期审计机制

| 频率 | 动作 | 责任人 |
|------|------|--------|
| 每周 | 快速核对（有代码变更时） | 开发者 |
| 每月 | 完整审计（逐模块对比） | 开发者 + 审查者 |
| 架构变更 | 强制审计（目录/模块关系变化） | 架构师 |

### R8: 先规划后执行（新增）

**背景**：本对话中反复出现"方向错了还在跑"的问题——grid search 跑在错误的参数上白费8小时、自进化集成在选股bug没修完之前。

**规则**：

```
每个新任务前必须经过:
  1. 花 2 分钟说清楚目标：“做什么 → 最优解长什么样 → 怎么验证做对了”
  2. 确认前置依赖：“有什么必须先修好，否则后面白做？”
  3. 用户确认后才动手
```

### R9: 长时间运行保护（新增）

1. **预计 > 2 分钟的任务，必须先问用户**：确认是否要跑
2. **定期检查方向**：长时间任务启动后先验证前 1-2 个中间结果，方向错了直接 kill
3. **明确停止条件**：开始前想好 "什么情况下这个结果是无用的"

### R10: 大道至简

**背景**：本对话中反复出现过度设计的倾向——贝叶斯优化、归因树、控制变量、多目标优化、决策树分裂，一个接一个，越搞越复杂。

**规则**：

```
选择方案时，永远问自己三个问题:
  1. 这个方案的最简形态是什么？（能不能砍到只剩核心逻辑）
  2. 多出来的复杂度，它能带来什么不可替代的价值？
  3. 如果明天我不维护这个项目了，新来的人看得懂吗？

如果答案不清晰 → 砍掉复杂度，回到最简方案。
```

**违反案例**：
- 参数优化从 Optuna → 归因树 → 控制变量 → 回到坐标下降法。绕了一大圈回到原点。
- 如果没有 R10，归因树那个版本可能就上线了——复杂、难理解、而且不比最简单的方案更好。

---

## 从 QuantSystem 学到的教训

1. **不要写两份文档**（设计文档 + 对照文档）。两份文档必然脱节。
2. **不要保留旧文档不做标记**。旧文档必须明确标记 DEPRECATED。
3. **不要只在脑子里同步文档**。文档不同步的问题不会自动消失。
4. **参数调优必须同步文档**。改了参数不改文档 = 埋雷。
5. **代码重构不改文档是最常见的疏忽**。重构意味着逻辑不变，但如果重构改变了接口或行为，文档必须更新。
