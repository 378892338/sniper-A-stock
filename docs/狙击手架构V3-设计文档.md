## 1. 架构总览

```
每日循环:
  ┌─ L0: 市场状态 ──→ 4 维合成 (趋势35%+量能25%+宽度25%+北向15%) → 0-100
  │   L0 ≥ 70: 满仓档开仓（目标总仓位50%, 最多5只）
  │   64 ≤ L0 < 70: 偏强档开仓（目标总仓位30%, 最多5只）
  │   L0 < 64: 不开新仓, 存量持仓自然退出 + 主动减仓到30%
  │
  ├─ L1: 板块排名 ──→ 4 维合成 (动量35%+资金25%+广度20%+热度20%) + Beta中性化
  │   输出: 统一 Top 3（2026-06-06 Strategy A 不再分层）
  │
  ├─ L2: 个股评分 ──→ 12 因子合成 (趋势+量能+MACD+RSI+资金流+大单+龙虎榜+
  │     EPS+ROE+营收增速+市值+换手率)
  │   输出: 每板块 Top 1（Top3板块 × 1只 = 3只候选）
  │
  ├─ L3: 入场过滤 ──→ 硬性条件（价格3-300、成交额>100万、换手率<30%、非涨停）
  │     + 柔性条件（L2评分≥79、板块排名≤3）
  │   输出: 候选个股
  │
  └─ L4: 退出链 ──→ 两层止损 + 辅助退出
       未盈利阶段: 日内最低价 ≤ -3% → 初始止损
       脱离成本后: 从最高点回撤 -5% → 动态止盈
       辅助退出: 跌破MA20 / 超时43天
```

## 2. L0 — 市场状态（MarketScorer）

**文件**: `sniper/layers/l0_market.py`
**设计文档对应**: `factor-design-v2.md` §2 (已废弃)

### 2.1 4 维评分

| 维度 | 满分 | 计算方法 | 权重 |
|------|------|---------|------|
| 趋势 | 0-100 | (close - MA20) / MA20 → -5%→0, 0%→50, +5%→100 | 35% |
| 量能 | 0-100 | volume_ratio (volume / MA20_vol) → 0.5→0, 1.0→50, 2.0→100 | 25% |
| 宽度 | 0-100 | 上涨家数占比 × 100 | 25% |
| 北向资金 | 0-100 | 近5日净流入均值 → ±50亿=±50分 | 15% |

### 2.2 开仓判定（2026-06-06 Strategy A 修订）

| 合成分 | 操作 |
|--------|------|
| **≥ 70** | 开仓（Top3板块，目标总暴露50%，最多5只） |
| **64-69** | 开仓（Top3板块，目标总暴露30%，最多5只） |
| **< 64** | 不开新仓，主动减仓到总仓位的30%（不等止损/止盈触发） |

**数据支撑**（2026-06-06 回测验证）：
- L0 ≥ 70: 胜率61.6%, 盈亏比3.48
- 64 ≤ L0 < 70: 偏强区间，30%仓位参与可捕捉更多机会
- L0 < 64: 胜率<45%, 中位PnL为负

> **2026-06-06 Strategy A 变更**：开仓门槛从 70 降至 64，新增 64-70 偏强区间 30% 仓位参与。
> 板块选取统一为 Top3（不再按 L0≥75/70 分层）。
> 8年回测验证：累计 570%（原 173%），夏普 2.58（原 1.47），回撤反而更低。

| 维度 | 满分 | 计算方法 | 权重 |

| 动量 | 0-100 | 近 N 日涨幅均值 → 横截面归一化 | 35% |

### 3.2 Beta 中性化（2026-06-03 新增）

在 composite_scores 中，对板块得分做 Beta 中性化：

```python
scores -= market_mean × 1.0  # 减去沪深300近期平均收益
```

简化实现：用沪深300近期平均收益做偏移。未来可升级为回归式 beta。

### 3.3 输出（2026-06-06 Strategy A 统一取前3）

```python
top_sectors(date, top_n=3) → list[str]: 板块名列表，统一取前 3（Strategy A 取消 L0 分层）
```

板块选取不再依赖 L0 分层，始终取前 3 个板块。

| 因子 | 满分 | 计算方法 | 权重 |
|------|------|---------|------|
| trend | 0-100 | MA10偏离: 50+(close-MA10)/MA10×200 | 15% |

**综合分 = Σ(factor × weight) / Σ(weights)**

### 4.2 双评分路径

L2 评分有两种路径，根据有无预计算因子文件自动选择：

```
top_stocks(date, sectors):
  1. 候选池: 从 L1 Top 板块收集个股 (get_sector_stocks)
  2. 路径判定:
     ├─ 有预计算因子文件 (outputs/precomputed/l2_factors/{date}.parquet) →
     │    加载因子 → 计算综合分 → 按分排序 → 取 Top N
     │    覆盖 ~2,546 只（历史文件，截至 2026-05-13）
     │
     └─ 无因子文件 → 实时计算:
           _preload_bars() 批量加载日线 (1次SQL)
           _compute_technical_factors_batch() 向量化算 8 因子
           _batch_fund_flow_factors() / dragon_tiger / financial
           合并 → 计算综合分 → **按分降序排序** → 取 Top N
           覆盖 ~4,970 只（随 daily_bars 更新）

  ⚠️ V3.7.1 (2026-06-06) Bug 修复:
     _score_batch_vectorized() 返回前缺少 .sort_values("score", ascending=False)，
     导致实时路径下 Top N 为字母序前 N 只（全是 000xxx），
     创业板(30xxx)和沪市(60xxx)被静默挤出。
     修复: .head(CFG.top_n) 前加 sort_values。
```

## 6. L4 — 退出链（ExitChain）

**文件**: `sniper/layers/l4_exit.py`
**设计文档对应**: `factor-design-v2.md` §5 (退出信号)

### 6.1 两层止损 + 辅助退出（2026-06-06 修订，2026-06-06 V3.6 新增分级退出）

**核心逻辑**：根据持仓是否脱离成本区间，采用不同的止损规则。**根据市场环境（L0评分）决定全退还是减半仓**。

```
评估每笔持仓:

先获取当日数据: 收盘价, 日内最低价, 持仓最高价, L0综合评分

阶段判断:
  ┌─ 未盈利阶段 (highest_price ≈ entry_price):
  │   检查: 日内最低价 ≤ entry_price × (1 + stop_loss)
  │   参数: stop_loss = -3%
  │   目的: 入场失败快速认错, 小亏离场
  │
  └─ 脱离成本后 (highest_price > entry_price):
      检查: (close - highest_price) / highest_price ≤ trailing_stop
      参数: trailing_stop = -5%（从最高点回撤）
      目的: 保护浮盈, 不让盈利变亏损

退出力度取决于市场环境（2026-06-06 Strategy A 三层）:
  ┌─ L0 ≥ 70: 止损/止盈触发时 → 减半仓(50%)，保留剩余仓位
  │   好处: 不因短期波动踏空强趋势
  ├─ 64 ≤ L0 < 70: 止损/止盈触发时 → 减到30%（卖70%留30%）
  │   好处: 偏强市场控制风险同时保留部分仓位
  └─ L0 < 64: 止损/止盈触发时 → 全仓退出
      目的: 市场偏弱时及时离场

辅助退出（所有阶段都检查，始终全退）:
  1. 跌破 MA20: close < MA20(20日均线) → 趋势转弱退出
  2. 超时退出: 持仓 ≥ 43 天 → 强制退出

主动减仓（2026-06-06 V3.7 新增 — 不等止损/止盈信号，主动降低暴露）:
  条件: L0 < 64 AND 总暴露 > 总资金的 30%
  动作: 按比例减仓至总暴露 = 30%（所有持仓等比例减仓）
  目的: L0 < 64 市场环境恶化，不等止损触发就主动减仓控制回撤
  └─ L0 64-69: 正常执行止损/止盈规则，不主动减仓
```

**优先级**（2026-06-06 Strategy A 三层退出）：

| 优先级 | 规则 | 适用范围 | 参数值 | L0≥70 | 64≤L0<70 | L0<64 |
|--------|------|---------|--------|-------|----------|-------|
| 1 | 初始止损（日内最低价） | 未盈利阶段 | -3% | **减半仓**(50%) | **减到30%**(卖70%) | 全仓退出 |
| 2 | 动态止盈（从最高点回撤） | 脱离成本后 | -5% | **减半仓**(50%) | **减到30%**(卖70%) | 全仓退出 |
| 3 | 跌破 MA20 | 所有阶段 | close < MA20 | 全仓退出 | 全仓退出 | 全仓退出 |
| 4 | 超时退出 | 所有阶段 | 43天 | 全仓退出 | 全仓退出 | 全仓退出 |
| 5 | 主动减仓（L0<64） | L0<64 | 总暴露≤30% | N/A | N/A | 等比例减仓 |

> 变更原因：归因分析显示牛市中止损过于频繁（47.7%的出场来自初始止损），
> 在强趋势环境中减半仓而非全退，既能控制风险又不踏空主升浪。

## 7. 回测引擎（BacktestEngine）

**文件**: `sniper/engine/backtest.py`, `sniper/engine/risk.py`

### 7.1 逐日模拟流程

```
for each trading_day:
    1. L0 状态分档 → 3档: ≥70(满仓档) / 64-70(偏强档) / <64(弱势档)
    2. L0 档位提醒 → L0在64-70偏强区间时提示关注仓位降至30%
    3. L4 退出检查 → 3层: ≥70减半仓 / 64-70减到30% / <64全平 + 辅助退出
    4. 风控检查 → 组合回撤5%强制平仓
    5. L0<64 主动减仓 → 总暴露减至30%（在风控之后、入场之前）
    6. L1 板块评分 → 统一 Top3（2026-06-06 取消分层）
    7. L2+L3 入场 → 候选 → 按目标暴露动态分配（≥70→50%, 64-70→30%）→ 开仓（仅 L0≥64）
    8. 日终更新 NAV (含停牌回退)
```

### 9.3 Fetcher 反检测层（2026-06-05 新增）

**文件**: `shared/fetcher.py`

Fetcher **不新建 ABC、不做源路由**，现有 `get_source()` 已处理源自动选择和健康路由。Fetcher 只做现有代码完全没有的：

#### 请求伪装

```
User-Agent 轮换: 每次请求从 20+ 主流浏览器 UA 池随机取
请求间隔: 对数正态分布（最像人类），均值 1.5s，标准差 0.5s
突发控制: 连续 5 次请求后强制停顿 3~8s
时段感知: 9:30~15:00 请求间隔不变，其他时段间隔 ×2
```

#### 核心流程

```python
class Fetcher:
    def fetch_stock_daily(self, symbol, start, end) -> pd.DataFrame:
        # 1. ST/920 过滤 → 直接返回空
        if symbol.startswith("920") or self._is_st(symbol):
            return pd.DataFrame()
        # 2. FetcherGuard 延时（反检测）
        self.guard.wait()
        # 3. 委托给现有工厂，不重复路由
        ds = get_source()
        return ds.fetch_daily(symbol, start, end)
```

### 9.4 Normalizer 格式统一层（2026-06-05 新增）

**文件**: `data/normalizer.py`

统一格式标准**不是"数据源能提供什么"**，而是**"因子需要什么"**：

```python
FACTOR_REQUIRED_FIELDS = ["open", "high", "low", "close", "volume", "amount"]
FACTOR_DESIRED_FIELDS = ["turnover"]
```

| 等级 | 字段 | 来源 | 说明 |
|------|------|------|------|
| P0 必须 | close, open, high, low | 所有源 | 无此数据该源不能用 |
| P0 必须 | volume | EM / Sina / Baostock | TX 源无 volume → 降级 |
| P0 必须 | amount | 所有源 | 单位换算后入库 |
| P1 应有 | turnover | EM 有 | ✅ 2026-06-05 新增列到 schema |
| P2 可选 | adj_factor | 未来扩展 | 复权校验用 |

关键变更：
- **缺失字段填 NaN，不填 0**
- **源不支持的字段不捏造**（TX 无 volume → volume=NaN → 触发降级）
- **normalize() 不做数据路由**，只做格式转换

### 9.5 Schema 自动对齐机制

```
因子开发者新增字段 X 到 FACTOR_REQUIRED_FIELDS
        │
        ▼
ensure_schema() 启动时自动检查
  ├─ daily_bars 表有 X 列？ → 无 → ALTER TABLE ADD COLUMN
  ├─ 现有数据 X 列全为 NULL？ → 触发补下
  └─ 数据源能提供 X？ → 更新 Normalizer.COLUMN_MAP
```

**文件**: `data/local/schema.py` — `DAILY_BARS_COLUMNS` 动态读取 `FACTOR_REQUIRED_FIELDS`
