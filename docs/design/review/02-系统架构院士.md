# 院士会审意见 #2 — 系统架构

**审查专家：** 分布式系统与数据架构 院士
**审查范围：** 卷 C（全卷）、卷 E

---

## 2.1 肯定项

### [A1-A] Journal-first 设计正确
方案将 PipelineJournal 作为整个管线的真相源（source of truth），PROBE 阶段依赖 journal 而非仓库数据做决策。仓库只有最终状态（当前最新日期），journal 保留了因果链，恢复场景中至关重要。

### [A1-B] intraday_snapshot 与 daily_bars 物理隔离正确
12:00 数据写入独立表，16:00 写入主表，两者通过 (symbol, date) 可关联互不污染。

---

## 2.2 否决级问题

### 〖A-CRIT-01〗流式模式的并发架构未定义

方案描述 "ThreadPoolExecutor 但仍保持流式" 但未明确并发模型。

**问题：**
```
现有模式：50 线程并发请求 → 5000×0.3s ≈ 30s
流式串行：1 线程闭环 → 5000×0.5s ≈ 41min（不可接受）
```

**必须采用：N 个 worker，每个内部串行闭环，worker 之间并发。**
```python
def run_parallel(self, symbols):
    with ThreadPoolExecutor(max_workers=self.n_workers) as pool:
        batches = self._chunk(symbols, self.n_workers)
        futures = [pool.submit(self._run_batch, batch) for batch in batches]
```

### 〖A-CRIT-02〗PROBE 决策应依赖 WRITE 记录而非 FETCH 记录

PK `UNIQUE(run_id, step, symbol)` 允许不同 step 同 PK。但如果 FETCH=ok 但 WRITE 失败（过程断电）→ Journal 中出现 FETCH 无 WRITE → PROBE 误判为已完成。

**修复：** PROBE 决策规则：
```
需要抓取 = NOT EXISTS(WRITE=ok AND data_end >= today)
```

### 〖A-CRIT-03〗StreamEngine 与 Fetcher 的双重并发域

`shared/fetcher.py` 已有降级链 + 重试 + HealthTracker + 内置并发。StreamEngine 如果再包一层并发，出现双重控制。

**修复：** StreamEngine 定义为编排层（orchestrator），不重新实现网络并发。fetch 委托给 Fetcher 或 mootdx source。

## 2.3 严重问题

### 〖A-CRIT-04〗DataSource 接口不支持 journal 回调

现有 `DataSource.fetch_daily()` 返回 DataFrame，无注入 journal 的钩子。

**修复：** StreamEngine._fetch_one() 包装调用，在外部写入 journal：
```python
def _fetch_one(self, symbol):
    t0 = time.time()
    try:
        df = self.source.fetch_daily(symbol, ...)
        self.journal.log(..., status="ok", elapsed_ms=...)
        return df
    except:
        self.journal.log(..., status="fail")
        return None
```

## 2.4 建议汇总

| 编号 | 建议 | 严重度 |
|------|------|--------|
| A-CRIT-01 | N worker × per-worker 串行闭环，可配置 n_workers | **阻断** |
| A-CRIT-02 | PROBE 依赖 WRITE 记录，非 FETCH | **阻断** |
| A-CRIT-03 | StreamEngine 只编排，不实现网络并发 | 高 |
| A-CRIT-04 | 外部包装 _fetch_one，不修改 DataSource 接口 | 高 |
| A-SUG-01 | intraday_snapshot 加 date 索引 | 中 |
| A-SUG-02 | n_workers 默认 5（兼顾速度 + 防封） | 中 |
