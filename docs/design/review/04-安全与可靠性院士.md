**审查专家：** 网络安全与系统可靠性 院士
**审查范围：** 卷 D、卷 C.5、历史教训对照

保留了 `health_tracker.record_failure/check_and_attempt_recovery` 机制，是从 baostock IP 封禁事故（2026-06-23）继承的正确设计。

---

## 4.2 否决级问题

### 〖S-CRIT-01〗方案完全未考虑 baostock 封禁的根因复现风险

**baostock 事故根因（2026-06-23）：**
```
多线程高频爬取（max_workers=5）+ 大规模全量扫描
→ 服务端检测异常流量模式 → IP 黑名单封禁
→ 333 个子进程 BSLOGIN_FAIL → serve_report 超时 → 全链路熔断
```

mootdx 的 TCP 7709 协议同样面向公共服务器（30+ 公开 IP），同样面临：
- 高频批量请求被检测为爬虫
- 单一出口 IP 被封导致全线不可用

**必须增加：**
1. mootdx 请求频率限制：≤ 1次/10秒（全市场 ≈ 5 次TCP请求/分钟）
2. 多 IP 轮换池：从 consts.py 的 30+ 服务器中随机选择
3. 如果连续 3 次失败 + HealthTracker 标记不可用 + 立即降级到 Tencent 源
4. **绝对不能**用 mootdx 做全量 5000 只并发日线下载

### 〖S-CRIT-02〗TCP 7709 明文协议无保护

通达信 TCP 7709 纯明文二进制协议，无加密、无签名、无认证。

**潜在风险：** 中间人可篡改价格数据，注入恶意响应。

**建议：** 文档声明：「通达信数据仅用于个人量化分析，不应用于交易决策」。

### 〖S-CRIT-03〗单一出口 IP 是单点故障

mootdx 的 30+ 服务器在客户端侧分散连接，但**服务端看到的永远是同一个出口 IP**。该 IP 被 ban = 全线不可用。

**缓解：** 无解（TCP 协议限制）。确保降级链（Tencent qt.gtimg.cn）始终可用。

---

## 4.3 严重问题

### 〖S-CRIT-04〗active1/active2 状态位被丢弃

源代码中 active1 和 active2 包含停牌/退市等状态标记。丢弃它们意味着 PROBE 阶段无法判断股票是否停牌，STOP 股依然会发起 TCP 请求浪费连接。

**建议：** 保留 active1 字段，停牌股（active1 有特定标记）在 PROBE 阶段直接 SKIP。

---

## 4.4 建议汇总

| 编号 | 建议 | 严重度 |
|------|------|--------|
| S-CRIT-01 | 请求频率限制 + IP 轮换 + 3次失败降级 | **阻断** |
| S-CRIT-02 | 明文协议声明（不用于交易决策） | 低 |
| S-CRIT-03 | 确保 Tencent 降级源始终可用 | 高 |
| S-CRIT-04 | 保留 active1 字段检测停牌 | 中 |
| S-SUG-01 | heartbeat=True 保活 | 中 |
