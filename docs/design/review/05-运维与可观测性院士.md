# 院士会审意见 #5 — 运维与可观测性

**审查专家：** 站点可靠性工程（SRE）与可观测性 院士
**审查范围：** 卷 C.3、卷 D、迁移路径

---

## 5.1 肯定项

### [R1-A] Journal 恢复语义设计完善
`get_missing_stocks()` 的 data_end vs today 比对逻辑能覆盖：中断恢复、增量补缺、重跑等全部场景。

---

## 5.2 否决级问题

### 〖R-CRIT-01〗Journal 无过期策略，无限增长

```
5000 只 × 5 step × 365 天 = 9,125,000 行/年
每行 ~200 字节 => ~1.8GB/年
```

仅靠 SQLite 无法高效查询。**必须增加：**
```python
def cleanup(self, keep_days=30):
    """1) 旧日志聚合为 daily_summary
       2) DELETE 原始行
       3) VACUUM"""
```

### 〖R-CRIT-02〗没有管道运行中的可见性接口

```
用户现在问：管道跑到哪了？还有多久？
现状：只能等 logger 输出，无实时查询
```

**增加：** `get_run_progress(run_id)` → `{total, done, ok, fail, progress_pct, eta}`

### 〖R-CRIT-03〗首次迁移路径未设计

旧管道无 journal 记录，首次切换到新管道时：
- 5000 只全部 PROBE 显示"需要抓取" → 全量重抓，耗时 ~40min
- 更糟：旧管道和新管道同时写入，无 journal 协调 → 数据冲突

**推荐迁移策略：**
```
步骤 1：部署新代码，创建 journal 表，旧管道继续运行（只读模式）
步骤 2：首批全量抓取（设置 skip_validation=True 加速）
步骤 3：确认 journal 覆盖完整，切换为新管线
步骤 4：删除旧锁文件和临时缓存
```

---

## 5.3 严重问题

### 〖R-CRIT-04〗现有 lock file 与新 StreamEngine 的兼容性

`run_pipeline.py:569-611` 有 PID 锁文件机制，但 StreamEngine 不自感知管道模式。lock file 区分 intraday/full，但 StreamEngine 只接收 mode 参数。

**建议：** lock file 机制**不动**，StreamEngine 从上层（run_pipeline.py）接收 mode 参数，不自感知模式。

### 〖R-CRIT-05〗Journal 表缺少 data_end 索引

`get_missing_stocks()` 和 `get_last_fetch()` 的查询模式是 `WHERE symbol=? AND step=? AND status=? ORDER BY data_end DESC`——没有 `data_end` 索引时，SQLite 对 900 万行做全表扫描排序。

**建议：** `CREATE INDEX idx_journal_lookup ON pipeline_journal(symbol, step, status, data_end);`

## 5.4 建议汇总

| 编号 | 建议 | 严重度 |
|------|------|--------|
| R-CRIT-01 | Journal cleanup() + 30 天归档 | **阻断** |
| R-CRIT-02 | get_run_progress() 进度接口 | 高 |
| R-CRIT-03 | 首次迁移全量重抓（skip_validation） | **阻断** |
| R-CRIT-04 | lock file 不动，StreamEngine 不自感知 | 中 |
| R-CRIT-05 | data_end 加索引 | 高 |
| R-SUG-01 | 增加 journal 写入 batched commit（≤10 条/批） | 中 |
