## 6.1 否决级问题

### 〖B-CRIT-01〗get_k_data() 的日期计算存在逻辑错误（影响现有 MootdxSource）

```python
# mootdx/quotes.py:428-455, get_k_data():
def get_k_data(self, code, start_date, end_date):
    first = (pd.to_datetime(end_date) - pd.to_datetime(datetime.now().date())).days
    first = (abs(first), 0)[first >= 0]  # ← 这里逻辑错误

    last = (pd.to_datetime(start_date) - pd.to_datetime(datetime.now().date())).days
    last = (abs(last), 0)[last >= 0]  # ← 这里逻辑错误
```

**分析：** `first` 和 `last` 的命名是**反直觉的**。按照通达信 API 的语义，`start` 参数表示"从最新数据往前推多少条"——这是一个偏移量，不是日期。但 `get_k_data` 试图将日期转换为偏移量：

```
假设 today=2026-06-26, end_date=2026-06-26:
first = (2026-06-26 - 2026-06-26).days = 0  # 正确：从最新数据开始

假设 today=2026-06-26, start_date=2020-01-01:
last = (2020-01-01 - 2026-06-26).days = -2368
     = abs(-2368) = 2368  # 正确：往前拉 2368 天

但 last 减去 int(last/3.5):
last -= int(last/3.5)  # = 2368 - 676 = 1692
# 减去非交易日估计值 → 近似 5 年交易日的正确天数
```

**问题场景：** 如果 `end_date < datetime.now()`（比如回补历史数据），`first` 的计算会错误：
```
today=2026-06-26, end_date=2020-01-01:
first = (2020-01-01 - 2026-06-26).days = -2368
     = abs(-2368) = 2368  # 从 2368 天前开始！而非 end_date 开始
```

**结论：** 现有 `MootdxSource.get_k_data()` 在获取非最新数据时（增量补缺），可能返回**多余的数据**（从更早开始）。这不影响数据完整性（有日期过滤），但浪费带宽。**无需立即修复，但应记录技术债。**

### 〖B-CRIT-02〗全局单例 Quotes 实例在多线程下存在响应串扰

```python
# data/sources/mootdx.py:49-59
def _get_client(self):
    if "std" not in self._instances:
        with self._instance_lock:
            if "std" not in self._instances:
                client = Quotes.factory(market="std", ...)
                self._instances["std"] = client
    return self._instances["std"]
```

**问题：** 这是一个全局单例连接。底层 `TdxHq_API` 使用 `self.lock`（线程锁）保护 `send_recv`，所以**不会**出现两个线程的数据交叉。

```python
# tdxpy/base_socket_client.py:
def send_recv(self, pkg):
    with self.lock:
        self.client.send(pkg)
        return self.client.recv()
```

所以线程安全是有保障的。**标记为安全，不是问题。**

### 〖B-CRIT-03〗bestip=False 导致连接可能不是最优

```
现有 MootdxSource: Quotes.factory(market="std", bestip=False)
=> 使用 config.get('SERVER').get('HQ')[0] = 深圳双线主站1

如果该服务器负载高或网络延迟大，没有自动切换到更优的服务器。
```

**建议：** 12:00 盘中管道宜设 `bestip=True`（启动时探测最优 IP，耗时 ~2-3 秒但收益大），全量降级备源保留 `bestip=False`（快速连接）。

## 6.2 严重问题

### 〖B-CRIT-04〗Python 3.13 + mootdx 0.11.7 的兼容性未验证

```
pyproject.toml: requires-python = ">=3.13"
mootdx 0.11.7 的最后更新年份？
```

mootdx 依赖 `tdxpy`，而 `tdxpy` 使用 `struct` 和 `cython` 风格的代码（注释中有 `# cython: language_level=3`）。虽然运行时不依赖 Cython（代码已编译），但 Python 3.13 的某些内部 API 变化（如 `datetime` 模块的 C 实现）可能导致意外行为。

**建议：** 编写最小探针验证 mootdx+tdxpy 在 Python 3.13 下的 TCP 收发能力。

### 〖B-CRIT-05〗流式管线中不同股票可能来自不同数据源，破坏了"全链路一致性"

```
现有假设：所有日线来自同一个数据源（如 jqdata）
→ 数据口径一致（复权/不复权一致）

流式管线：股票 A 来自 mootdx（不复权），股票 B 来自 jqdata（前复权）
→ 跨股票因子计算（如相对强弱）口径不一致
```

**影响：** 如果因子计算涉及跨股票比较（排序、百分位、配对交易），不同复权策略导致虚假信号。

**修复：** 
1. `DATA_SOURCE_PREFERENCE` 按源分组，而不是按股票分组
2. 或：FETCH 阶段只使用降级链的**第一个可用源**，所有股票统一
3. 或：数据归一化层（normalizer）统一做复权处理

**建议：** 流式管线的源选择策略改为**"统一的降级链"**而非"每只股票独立选源"。

### 〖B-CRIT-06〗缺失 sector/industry 数据的流式验证

方案只验证了 `daily_bars` 的流式处理。但 `update_market_indices()`、`update_sw_indices()`、`stock_list`、`trade_calendar` 仍然使用**旧的批量方式**。这部分数据占 pipeline 时间 ~20%，但在方案中没有对应的流式处理。

**这不是问题：** 指数数据量小（十几只），全量更新只需几秒，流式化的收益极低。但 Journal 应该**也记录**这些轻量表的更新，确保全链路可追溯。

## 6.3 边界场景

### 〖B-EDGE-01〗闰秒/时区/夏令时影响

- 通达信服务器时间 = 北京时间（UTC+8）
- 系统时间可能是 UTC+8（中国环境）
- 如果系统运行在 UTC 时区的服务器上，时间偏移会导致 12:00 的判断偏差

**解决方案：** 时间比较使用 `datetime.now(tz=timezone(timedelta(hours=8)))` 强制北京时间。

### 〖B-EDGE-02〗非交易日 12:00 的 mootdx 连接行为

非交易日通达信服务器依然在线（可 TCP 连接），但返回的数据中：
- `price = last_close`（未更新）
- `vol = 0`（无成交）
- `open = high = low = price`

**PROBE 阶段应该使用交易日历提前跳过非交易日**，避免不必要的 TCP 连接。

### 〖B-EDGE-03〗新股上市首日的数据问题

新股上市首日：
- `last_close` = 发行价（可能为 NaN）
- `price` 与 `last_close` 的差值无意义
- `active1` 状态位可能标记为特殊状态

**PROBE 阶段应过滤上市 < 5 天的股票**（与现有 EXCLUDE_NEW_IPO 逻辑一致）。
